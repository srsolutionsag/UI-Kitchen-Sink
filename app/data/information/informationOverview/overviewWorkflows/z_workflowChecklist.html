If the workflow for adding/updating/importing instructs to implement a new component for the UI-Kitchen-Sink, the following points must be checked:

Point 1, JSON:
Make sure there does not exist already a UI-Kitchen-Sink Element that can be used for the targeted use case. You MUST create a new entry if the DOM structure needs to be changed for the specific component only, new less code or component specific JS needs to be added. If the change is usable for all existing objects of an entry, then you SHOULD just change the existing one. If this is not the case you MUST apply the following rules:
  • Check for the right category in the kitchen sink for your new component. If you think, a new category needs to be created, propose it the the UI-Kitchen-Sink group along with a description of this new category.
  • You MUST use the 'UI-Kitchen-Sink Template' to create your pull request for the UI-Kitchen-Sink.
  • Chose an id for your new component. You MUST make sure, the id does not already exist.
  • You SHOULD chose an id, describing the functionality of your component well. If possible, look for a naming that already exists (such as Modal, Popover, etc.).
  • If your component is a special version of an existing one, you MUST set the id to [existingComponentNewFunctionality]. Example: modalWithForm. You MUST set the relation of the new component with the 'isA' attribute.
Point 2, Bootstrap:
Check on Bootstrap 3 if there exists a Bootstrap component that could be used or just slightly modified.
Point 3, DOM:
Write your own DOM structure. Be sure to follow the guidelines concerning accessibility.
Point 4, LESS:
Write your own less. You MUST apply the following rules:
  • Check if there are existing bootstrap or ILIAS less classes that could be used or extended.
  • Check if there are existing bootstrap or ILIAS less mixins that could be used.
  • Less variables presented in the UI-Kitchen-Sink SHOULD be used if they fit the context of a given class.
  • You MUST check if variables exists for often used attributes such as color or padding.
  • Be sure to design your classes mobile-first, meaning first define the class for a mobile device using a media query and in a second step add the exceptions for a bigger screen.
  • Components MUST have an outermost DOM element with at least the following classes: [il-id-of-parent-1-if-any] [il-id-of-parent-2-if-any] [il-id-component]. Example: il-dialog il-modal il-modal-with-form
  • New classes must precede bootstrap classses.
  • You SHOULD not use id's for styling. Only use id's for aria-roles and labels or js.
  • id's MUST be named in lower camel case. Example: ilModalTitle.
  • id's MUST be unique.
  • id's MUST proceed any other attributes, followed by classes.
  • If custom classes or variables are needed, be sure to follow the naming conventions for less variables and classnames.
Point 5, JS:
You SHOULD try to avoid JS as much as possible. Often JS code can be avoided by using attributes such as data-toggle and data-target offered by bootstrap approprietly. See Table Filter as example. If you need to implement JS, you MUST respect the JS Guidelines.
